# Оптимизация Ollama, контекст, кэширование и буферизация

## 1. Контекст не растёт по ходам

Полная история не передаётся. Размер запроса определяется текущим состоянием:

```text
статические правила
+ текущие объекты
+ рекомендация
+ legalActions
```

Количество объектов физически ограничено картой.

## 2. Кэширование контекста

Кэширование не означает хранение игрового мира внутри модели. Оно означает повторное использование уже вычисленного совпадающего префикса токенов.

Если запросы начинаются одинаково:

```text
общие правила
+ формат ответа
+ правила роли
```

движок модели может повторно использовать вычисления для этого префикса.

## 3. Условия хорошего совпадения

- одинаковая модель;
- одинаковый system prompt;
- одинаковый порядок разделов;
- стабильные имена полей;
- стабильный порядок ключей JSON;
- статические правила в начале;
- динамическое состояние в конце.

## 4. Keep alive

```json
{"keep_alive": "30m"}
```

`keep_alive` удерживает веса модели в RAM/VRAM и уменьшает задержку повторной загрузки. Это не игровая память.

## 5. Не передавать только дельту

Не следует писать:

```text
В прошлом запросе была карта.
Изменения: юнит №4 переместился.
```

Кэш не является гарантированной сессией. Полная актуальная карта мала и должна передаваться каждый раз.

## 6. Что реально создаёт нагрузку

- слишком длинные инструкции;
- огромный `legalActions`;
- reasoning-ответы;
- `think:true` с недостаточным лимитом;
- постоянная загрузка разных моделей;
- последовательные рапорты.

## 7. DeepSeek и think

Наблюдавшаяся проблема: reasoning съедает весь `num_predict`, а финальный JSON не появляется.

Меры:

- увеличить `num_predict`;
- сократить промпт;
- ограничить задачу текущим ходом;
- проверять наличие финального JSON;
- делать retry;
- отключить `think` у юнитов, закупки и рапортов.

## 9. Метрики

Сохранять:

- `prompt_eval_count`;
- `prompt_eval_duration`;
- `eval_count`;
- `eval_duration`;
- полную длительность;
- имя модели;
- роль агента;
- число объектов;
- число legalActions;
- число retry.

## 10. Буферизация

В контексте этого проекта полезнее говорить об очередях:

- очередь запросов;
- очередь рапортов;
- накопление событий до обновления UI;
- ограничение параллелизма.

Двойная и тройная буферизация — классический термин графики: front buffer показывается, back buffer рисуется, а при тройной буферизации есть дополнительный свободный буфер. Для AI это не обязательная технология.

## 11. Профилирование прежде оптимизации

Первая версия должна быть понятной и полной. После измерений можно сокращать поля, уменьшать `num_ctx`, отключать рапорты, агрегировать дальние объекты и менять модели.
