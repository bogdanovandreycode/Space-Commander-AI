# Валидация, тестирование и план интеграции через Codex

## 1. Типовые ошибки LLM

- неверный JSON;
- Markdown вокруг JSON;
- выдуманный `actionId`;
- ссылка на уничтоженный объект;
- путаница RED, BLUE и GRAY;
- попытка разведки при отсутствии тумана;
- пустой ответ;
- timeout;
- CORS;
- reasoning без финального JSON.

## 2. Парсинг JSON

1. `JSON.parse`;
2. извлечь блок ```json;
3. извлечь текст от первой `{` до последней `}`;
4. при неудаче применить fallback.

## 3. Валидация

### Штаб

- только существующие AI-юниты;
- без повторов;
- координаты внутри карты;
- приоритет 0–100;
- бюджет не выше кредитов;
- отсутствующие рекомендации заполняются стандартными.

### Юнит

Главное правило:

```text
actionId должен присутствовать в refreshed legalActions
```

### Закупка

- ID существует;
- хватает денег;
- резерв соблюдён;
- планета доступна;
- ремонтная блокировка учтена.

## 4. Fallback score

```text
score =
  orderFit
  + strategicValue
  - risk
  + 100 при уничтожении цели
  + 120 при колонизации
  + 50 при ремонте
  - 500 при гибели своего юнита
  - 250 при смертельной угрозе следующего хода
```

## 5. Обязательные тесты

1. Колонизатор не используется как разведчик.
2. При отсутствии нейтральной планеты колонизатор не строится.
3. Юнит отклоняет атаку, если рядом фрегат и контратака смертельна.
4. После уничтожения общей цели следующий юнит получает обновлённую карту.
5. Два юнита не занимают одну клетку.
6. Закупка не блокирует ремонтную планету.
7. Неверный `actionId` не исполняется.
8. Timeout штаба запускает fallback.
9. Markdown JSON корректно извлекается.
10. `think:true` без финального JSON приводит к retry или fallback.

## 6. Логирование

В debug-режиме сохранять payload, модель, сырой ответ, parsed JSON, warnings, фактическое действие, длительность, токены и причину fallback.

## 7. Пошаговое ТЗ для Codex

### Этап 1. Подготовка

- сделать резервную копию `civ3.js`;
- найти массивы юнитов и планет;
- найти кредиты, номер хода, движение, бой, колонизацию, ремонт, покупку и проверку победы;
- не переписывать отрисовку и управление игроком.

### Этап 2. Семантические классы

Сохранить старые `type`, добавить `getSemanticClass(type)`. Обязательно:

```text
scout → COLONY_SHIP
mapFullyVisible=true
fogOfWar=false
scoutingMechanic=false
```

### Этап 3. LegalActions

Реализовать:

```js
generateLegalActionsForUnit(unit);
```

Каждое действие должно содержать ID, тип, цель, риск, `orderFit`, `strategicValue`, `predictedResult` и `strategicTags`.

### Этап 4. Угрозы

Реализовать:

```js
getThreatenedCells(unit);
```

Учитывать движение, диагонали, кулдауны, препятствия и способности.

### Этап 5. Штаб

Создать `requestHeadquartersPlan()` и валидацию ответа.

### Этап 6. Очерёдность

Создать `normalizeExecutionOrder()`. После каждого действия обновлять карту.

### Этап 7. Закупка

Реализовать:

```js
generateLegalPurchaseActions();
requestProcurementDecision();
executePurchaseAction();
```

Передавать ремонтные намерения.

### Этап 8. Локальные агенты

Реализовать:

```js
buildLocalTacticalState(unit, radius);
requestUnitDecision();
validateUnitDecision();
executeUnitAction();
```

Начальный `tacticalRadius = 4`.

### Этап 9. Ремонт

Зафиксировать точные правила ремонта и отразить их в `legalActions`.

### Этап 10. Рапорты

После `actualResult` вызывать `requestUnitReport()` и хранить не более 30 рапортов на юнит.

### Этап 11. Ollama

Использовать `POST http://localhost:11434/api/chat`, `stream:false`, `format:"json"`, `keep_alive`, timeout и отдельные модели по ролям.

### Этап 12. CORS

Если игра открыта с `https://wp.local`, разрешить origin в Ollama и показывать понятную ошибку о `OLLAMA_ORIGINS`.

### Этап 13. Конфигурация

```js
const CIV3_AI_CONFIG = {
  headquartersModel: 'deepseek-r1:8b',
  procurementModel: 'gemma3:4b',
  unitModel: 'gemma3:4b',
  reportModel: 'gemma3:4b',
  tacticalRadius: 4,
  keepAlive: '30m',
  reportsEnabled: true,
  fallbackEnabled: true,
  debug: true
};
```

### Этап 14. Оркестратор

Итоговая точка входа:

```js
await multiAgentAi.runAiTurn();
```

## 8. Критерии готовности

- штаб выдаёт план;
- порядок реально соблюдается;
- карта обновляется после каждого действия;
- юнит может отклонить опасную рекомендацию;
- неверный ID не исполняется;
- закупка учитывает ремонт;
- колонизатор не используется как разведчик;
- рапорты основаны на фактах;
- при ошибке игра не зависает;
- старый AI доступен как fallback;
- UI показывает текущий этап AI-хода.

## 9. Что не делать

- не давать LLM прямой доступ к состоянию;
- не исполнять придуманные координаты;
- не хранить весь чат;
- не отправлять только дельту карты;
- не запускать все решения по старому снимку;
- не заставлять 4B-модель считать сложный бой;
- не считать prompt cache постоянной памятью.
