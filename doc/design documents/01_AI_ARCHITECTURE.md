# Архитектура многоагентного AI

## 1. Цель

AI строится как командная структура:

```text
Turn Orchestrator
├── Headquarters Agent
├── Procurement Agent
├── Unit Agent #1
│   ├── Decision
│   └── Report
├── Unit Agent #2
│   ├── Decision
│   └── Report
└── Optional Turn Summary Agent
```

Такая схема нужна не только ради силы AI, но и ради экспериментов: можно смотреть отдельные решения, приказы и рапорты, сравнивать модели и постепенно добавлять характеры и память экипажей.

## 2. Роли

### 2.1. JavaScript-движок

Движок является единственным владельцем состояния и отвечает за:

- координаты и занятость клеток;
- дальность хода и диагонали;
- правило движения раз в два хода;
- урон, броню и бонусы против классов;
- мгновенное уничтожение отдельных типов;
- колонизацию;
- ремонт;
- стоимость и место строительства;
- проверку победы;
- повторную проверку любого выбранного действия.

LLM не должна самостоятельно вычислять правила боя и не должна менять массивы состояния напрямую.

### 2.2. Штаб

Штаб видит глобальное состояние текущего хода и должен:

- оценить баланс сил;
- выбрать доктрину текущего хода;
- определить приоритетные цели;
- назначить каждому юниту рекомендацию;
- задать порядок обработки юнитов;
- указать допустимые альтернативы;
- отправить повреждённые корабли на ремонт;
- сформировать директиву закупки.

Штаб не должен строить маршруты на 3–4 хода вперёд и микроконтролировать каждую клетку.

### 2.3. Агент отдельного юнита

Каждый корабль обрабатывается отдельным запросом и получает:

- свой ID и семантический класс;
- текущие HP и состояние;
- компактные правила класса;
- рекомендацию штаба;
- локальную область карты;
- угрозы следующего хода;
- ближайшую поддержку;
- ремонтные планеты;
- события текущего AI-хода;
- список `legalActions`.

Агент выбирает один `actionId` и может выполнить рекомендацию, выполнить её частично, отложить, отступить, пойти на ремонт, занять лучшую позицию или ждать.

### 2.4. Агент закупок

Закупщик не переопределяет глобальную стратегию. Он выбирает конкретные допустимые покупки, учитывая кредиты, резерв, состав флотов, угрозы планетам и планы ремонта.

### 2.5. Агент рапорта

Рапорт формируется только после фактического исполнения. Он описывает реальный результат и не влияет на механику.

## 3. Разные модели

Рекомендуется разделить модели по ролям:

```text
Штаб:
- DeepSeek reasoning-модель;
- крупный Qwen;
- облачная модель через OpenRouter.

Юниты:
- Gemma 4B;
- другая быстрая локальная instruct-модель.

Закупка:
- Gemma 4B или отдельная недорогая модель.

Рапорты:
- та же локальная модель с большей temperature.
```

Сильная модель вызывается один раз за ход, а большое число дешёвых запросов выполняется локально.

## 4. Рекомендации вместо приказов

Хороший формат:

```json
{
  "unitId": 7,
  "recommendation": "PRESSURE_ENEMY_INTERCEPTOR",
  "targetType": "UNIT",
  "targetId": 12,
  "priority": 85,
  "acceptableAlternatives": [
    "TAKE_ADVANTAGEOUS_POSITION",
    "RETREAT_AND_REPAIR"
  ]
}
```

Локальный агент может отклонить рекомендацию, если атака ведёт к очевидно смертельной контратаке.

## 5. Очерёдность

Штаб возвращает `executionOrder`, например:

```json
{"executionOrder": [7, 5, 8, 4]}
```

Движок обязан нормализовать порядок: удалить неизвестные и повторяющиеся ID, пропустить уничтоженные юниты и добавить пропущенные в конец по стандартной инициативе.

## 6. Последовательное исполнение

Юниты не выбирают действия одновременно по одному старому снимку карты. После каждого действия карта обновляется, и следующий юнит получает новое состояние. Это предотвращает попытки атаковать уже уничтоженную цель, занимать одну клетку двумя кораблями и идти к уже захваченной планете.
